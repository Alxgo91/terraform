§ 01

Le modèle de responsabilité partagée

La sécurité est répartie clairement entre AWS et vous. Chacun son périmètre — et en cas d'incident, on sait qui est responsable.

Responsabilité AWS

Sécurité DU cloud

L'infrastructure qui fait tourner le cloud.

  • Datacenters & matériel physique
  • Réseau physique & hyperviseur
  • Maintenance de l'infrastructure

Votre responsabilité

Sécurité DANS le cloud

Tout ce que vous mettez et configurez dedans.

  • Vos données & leur chiffrement
  • Utilisateurs & permissions (IAM)
  • Pare-feu : Security Groups
→ VOUS Mot de passe faible, bucket S3 laissé public, données piratées : c'est votre configuration, donc votre responsabilité.
→ AWS Panne matérielle dans un datacenter : c'est l'infrastructure, donc à AWS de gérer.

Moyen mnémo : sécurité DU cloud = AWS · sécurité DANS le cloud = vous.

§ 02

Qu'est-ce qu'IAM ?

Le « videur » du compte AWS 100 % GRATUIT

IAM (Identity and Access Management) gère les identités et les autorisations : il permet de créer des utilisateurs, de les regrouper et de définir précisément ce que chacun peut faire. Avant chaque action sur un service, c'est lui qui contrôle l'entrée.

§ 03

Les 4 entités d'IAM

Quatre briques à connaître par cœur — trois identités, et le document JSON qui leur donne (ou refuse) des droits.

UtilisateurUser

Une identité pour une personne ou une application, avec ses propres identifiants.

1 collaborateur = 1 utilisateur. Jamais de compte partagé.

GroupeGroup

Un ensemble d'utilisateurs qui reçoivent les mêmes permissions en une seule fois.

Bien plus simple que de gérer les droits un par un.

RôleRole

TEMPORAIRE

Des permissions endossées le temps d'une tâche par un utilisateur, une appli ou un service AWS — sans identifiants permanents.

Idéal pour donner des droits à une instance EC2 ou à un service.

PolitiquePolicy

JSON

Le document JSON qui décrit les autorisations : quelles actions sont permises ou refusées, sur quelles ressources.

S'attache à un utilisateur, un groupe ou un rôle.

§ 04

Bonnes pratiques de sécurité

Trois notions clés, cinq réflexes à appliquer dès la création du compte.

Compte root LE PROPRIÉTAIRE

Créé à l'ouverture du compte, il possède tous les droits. À réserver aux tâches exceptionnelles — jamais pour l'usage quotidien.

MFA MULTI-FACTOR AUTHENTICATION

Mot de passe + code temporaire (appli mobile). Protège le compte même si le mot de passe est volé.

Moindre privilège LEAST PRIVILEGE

N'accorder que les permissions strictement nécessaires à la tâche, rien de plus. Réduit fortement les risques en cas de compromission.

✓ Checklist dès le jour 1

  1. Sécuriser le root : mot de passe fort + MFA, puis ne plus l'utiliser au quotidien.
  2. Créer un utilisateur IAM personnel pour l'administration courante.
  3. Appliquer le moindre privilège via des politiques précises.
  4. Regrouper les droits avec des groupes ; utiliser des rôles pour les services.
  5. Activer la MFA pour tous les utilisateurs sensibles.
Point de vigilance

Jamais de clés d'accès sur le compte root, et jamais d'identifiants partagés entre plusieurs personnes : en cas d'incident, impossible de savoir qui a fait quoi — c'est la perte de traçabilité.
